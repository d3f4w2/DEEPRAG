# 03. 系统架构与主链路

这一章的目标是：

当面试官说：

`你给我讲一下整体架构。`

你不会脑子一片空白，也不会只会说“前端 React，后端 FastAPI”。

你要做到的是：

1. 先讲模块
2. 再讲数据怎么流
3. 最后讲每个模块为什么这么设计

---

## 一、先背这个总结构

你可以这样开口：

`整体上这个系统分成四层：前端交互层、后端编排层、知识库与索引层、模型与评测层。前端负责问答与多模态入库，后端负责检索编排、证据判断和接口暴露，知识库层负责文档与摘要索引，模型层负责生成、工具调用和评测。`

这是最稳的开场。

---

## 二、架构拆解

### 1. 前端层

前端不是重点，但必须知道它的职责。

你可以说：

`前端是 React + TypeScript + Vite，主要承担三个交互界面：Chat、语音入库、图片入库。Chat 页负责展示消息、工具调用轨迹、Judge 状态和预算信息；语音/图片页负责把新信息做成可入库的知识。`

前端的面试价值在于：

1. 不是只有后端命令行
2. 可以可视化工具调用和证据链路
3. 说明你考虑了展示和可解释性

### 2. 后端层

这是你最该熟的部分。

可以说：

`后端是 FastAPI，核心是一个聊天编排器。它接收用户问题后，不是直接把问题丢给模型，而是先进行路由与检索规划，再调用 `search_paths` 和 `retrieve_sections`，整理 evidence_pool，然后交给 critic 判断是否足以回答。`

你要强调：

1. 后端不是简单转发 API
2. 有明确的编排逻辑
3. 有约束和判断

### 3. 知识库与索引层

这里要突出“结构化”和“可检索”。

可以说：

`知识库本身是文件结构，核心思想是保留原始结构。系统会基于 `summary_demo.json` 和文件路径生成轻量索引，`search_paths` 先在这个层面筛候选范围，再到正文里做片段抽取。`

这里面隐含两个点：

1. 文件结构本身就是知识的一部分
2. 检索不是一上来就读正文

### 4. 模型与评测层

可以说：

`模型层负责两类事情：一类是问答生成与工具调用，另一类是评测，比如 token、延迟、历史 RAGAS run。也就是说，模型不仅参与回答，也参与系统质量的观测。`

---

## 三、主问答链路怎么讲

最稳的方式，是按顺序讲。

### Step 1. 用户提问

用户从 Chat 页面输入问题。

比如：

`除了 AMOLED 和 OLED 还有哪些显示类型？`

### Step 2. 路由与检索规划

后端先决定这类问题大概需要多宽的检索范围。

你可以说：

`系统会结合 query 做轻量 route planning，决定 top_k、每文件抽几个 section、最少覆盖几个文件。`

### Step 3. `search_paths`

这一步不是读正文，而是找候选文件。

你可以说：

``search_paths` 基于路径和摘要做轻量候选检索，返回的是候选文件路径、分数和摘要。`

### Step 4. `retrieve_sections`

这一步从候选文件中抽证据。

你可以说：

`随后 `retrieve_sections` 会按 Markdown 标题切段，从候选文件中抽取高相关的证据片段，而不是整文件注入。`

### Step 5. 构建 evidence_pool

这一步是可信性的关键。

你可以说：

`系统会把检索到的标准化片段整理成 evidence_pool，作为后续回答的依据。`

### Step 6. critic 判断

这一步说明你不是“检索完就答”。

你可以说：

`critic 会判断当前证据是否足够支持答案，如果不够，就继续检索；如果仍然不足，就拒答，而不是硬编。`

### Step 7. 输出最终答案

最终答案必须带证据。

你可以说：

`最终输出不是单纯一段文字，而是回答正文加 `### 证据`，保证来源可追溯。`

---

## 四、语音和图片入库链路怎么讲

面试官如果问“语音和图片是怎么接进去的”，你可以这么说。

### 1. 都是双阶段

先 `draft`，再 `ingest`。

为什么这样设计：

1. AI 先给草稿
2. 人来确认
3. 减少脏数据直接入库

### 2. 语音链路

你可以说：

`语音页先用浏览器语音识别拿到实时转写，然后调用 `/voice/draft` 生成润色文本和摘要。用户确认之后再调用 `/voice/ingest`，写入 `Knowledge-Base/Voice-Notes/...` 并更新摘要索引。`

### 3. 图片链路

你可以说：

`图片页先上传图片，后端通过 OCR 和视觉模型生成摘要、描述、标签和检索关键词。确认之后写入图片 Markdown 笔记，同时保存原图路径，并同步更新索引。`

### 4. 为什么这很重要

重点不是“多模态很酷”，而是：

1. 新知识能进入统一问答链路
2. 入库后立即可检索
3. 说明系统具备扩展能力

---

## 五、评测链路怎么讲

这一段能拉开你和普通项目的差距。

你可以说：

`我还补了评测管理这一层，能对数据集发起评测任务，生成明细、汇总和报告。当前我主要看 40 题全量跑通情况、平均 token、平均延迟，以及历史兼容 run 的 RAGAS 可测结果。`

这段话能说明：

1. 你有质量意识
2. 你不是只看页面效果
3. 你知道指标也有口径问题

---

## 六、白板讲法

如果让你手画架构，照这个顺序画：

1. 左边画 `Frontend`
2. 中间画 `Backend / Orchestrator`
3. 下面画两个工具：`search_paths`、`retrieve_sections`
4. 右下画 `Knowledge Base + summary_demo.json`
5. 右上画 `LLM Provider`
6. 后面再补 `Critic / Evaluation`

你讲的时候按下面顺序：

1. 用户问题从前端进来
2. 后端做 route planning
3. 工具做候选路径检索
4. 再做证据片段抽取
5. evidence_pool 交给 critic
6. 最终输出答案与证据

这个顺序非常顺，几乎不容易乱。

## 七、面试官可能继续追问什么

### 追问 1：为什么不直接整文件召回？

回答：

`因为整文件召回上下文太大，信息密度不高，既增加 token 成本，也容易把模型注意力分散。两阶段检索的目标就是先缩小范围，再抽高价值证据。`

### 追问 2：为什么要保留文件结构？

回答：

`因为很多问题依赖全局结构和跨文件关系。文件结构本身就是知识的一部分，完全切碎后会损失这种信息。`

### 追问 3：critic 是必须的吗？

回答：

`不是所有项目都必须，但对这个项目来说很重要，因为我希望回答不是“像对的”，而是“有证据支撑的”。critic 的作用就是控制回答权限。`

### 追问 4：多模态是不是独立系统？

回答：

`不是，我刻意把它们设计成统一知识入口。无论语音还是图片，最终都会变成可检索的知识文件，复用同一条检索链路。`
