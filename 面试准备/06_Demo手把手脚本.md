# 06. Demo 手把手脚本

这一章的目标，是让你做到：

1. 知道 Demo 演什么
2. 知道每一步说什么
3. 知道出问题时怎么救

请记住一句话：

`Demo 的目标不是秀页面，而是证明“问题 -> 检索 -> 证据 -> 回答”这条主链路是真的。`

---

## 一、Demo 前的总原则

只演最稳的，不演最花的。

优先级顺序：

1. `证据驱动问答`
2. `工具链路可解释`
3. `语音/图片入库后可检索`

如果时间只有 1 分钟，只演前两个。

如果时间有 3 分钟，再补语音或图片。

---

## 二、演示前准备清单

面试前 10 分钟必须确认：

1. 后端已启动
2. 前端已启动
3. 页面能打开
4. 模型 provider 可用
5. 主问题能稳定回答
6. 语音/图片功能至少有一个是稳的

建议你面试前用这两个命令确认：

```bash
python -m pytest -q
```

```bash
cd frontend
npm run build
```

它们通过，会让你心态更稳。

---

## 三、最稳的 60 秒 Demo

### 0s - 10s：先定调

你先说：

`我给您快速看一下这个项目最核心的地方，不是它能回答问题，而是它能基于证据回答问题。`

这句话非常重要。

它一下就把面试官注意力从“聊天机器人”拉到“证据驱动系统”。

### 10s - 25s：输入主问题

建议主问题：

```text
除了 AMOLED 和 OLED 还有哪些显示类型？
```

你边输入边说：

`这个问题比较适合检验系统是否真的在做结构化检索，而不是只做相似文本匹配。`

### 25s - 40s：展示工具链路

当页面开始显示检索轨迹时，你说：

`这里会先走 `search_paths` 找候选文件，再走 `retrieve_sections` 抽证据片段，而不是一上来整文件喂给模型。`

如果页面展示了 Judge 或证据轨迹，就继续说：

`回答不是直接放行，而是要经过证据池和 judge 判断。`

### 40s - 55s：展示答案与证据

你指着答案里的证据块说：

`最终回答必须带 `### 证据`，所以我能追溯它引用了哪些文件、哪些片段。`

### 55s - 60s：一句话收尾

你说：

`所以这个项目的核心不是“回答出来”，而是“有证据地回答出来”。`

---

## 四、3 分钟增强版 Demo

如果面试官愿意多看一点，你可以用增强版。

### 第一段：先演主链路

还是先问主问题，证明：

1. 有检索
2. 有证据
3. 有可解释性

### 第二段：切到语音入库或图片入库

建议优先演语音，因为它更容易讲清楚“新知识进入系统”。

你可以说：

`除了问答，我还把语音做成了统一知识入口。这里不是录完就结束，而是先生成草稿，人工确认后再入库。`

然后简短操作：

1. 输入或录入一小段文本
2. 点击同步 AI 草稿
3. 点击确认入库

### 第三段：回到 Chat 验证刚入库内容

你问：

```text
刚才这条语音记录的核心内容是什么？
```

然后说：

`它会沿用同一条检索链路去命中新入库内容，而不是走另一套“特殊逻辑”。`

### 最后收尾

`我希望这个系统不是静态知识库，而是一个能持续吸收新知识的证据驱动问答系统。`

---

## 五、备选问题

主问题如果不稳，你必须准备备胎。

### 备选 1

```text
SW-2100-Flagship 是什么定位？请给证据。
```

适合展示：

1. 普通检索
2. 文件命中
3. 证据引用

### 备选 2

```text
Southwest Region 有多少门店？请给证据。
```

适合展示：

1. 数值问题
2. 证据驱动回答

### 备选 3

```text
刚入库的语音记录核心说了什么？
```

适合展示：

1. 入库后可检索
2. 同一链路复用

## 六、如果 Demo 出问题，怎么救场

这部分特别重要。

### 情况 1：模型响应慢

你说：

`这个项目当前也把 token 和延迟作为评测指标在看，所以如果现在慢一点，其实和我观察到的全量 run 行为是一致的。`

这样你把“卡顿”变成“可解释的现象”。

### 情况 2：语音识别不稳定

你立刻切手动输入，不要硬撑。

你说：

`语音页支持实时识别，也支持手动输入，我这里直接用手动文本来演示后续草稿和入库闭环。`

### 情况 3：图片模型波动

你就不演图片，直接回到主问答链路。

你说：

`图片我这边也做了统一入库链路，不过为了节省时间，我先聚焦展示这个项目最核心的证据驱动问答主链路。`

### 情况 4：主问题回答不理想

立刻切到备选问题 1。

不要解释太多，不要在现场 debug 太久。

---

## 七、Demo 时应该说的话

Demo 时最怕两种问题：

1. 一直点页面，不说人话
2. 一直说抽象概念，不落到页面

你要做到“动作一句，解释一句”。

推荐话术：

1. `这里先做候选路径检索。`
2. `这里再抽证据片段，不是整文件召回。`
3. `这里 Judge 会判断证据够不够。`
4. `这里最终答案必须带证据。`
5. `这里入库后会更新索引，下一轮就能搜到。`

---

## 八、千万不要这样 Demo

1. 一上来先讲前端样式
2. 花 1 分钟介绍按钮
3. 现场改代码
4. 现场解释一堆你自己都不确定的数据
5. 因为一个功能波动，把整个 Demo 节奏打乱

你要记住：

`主链路优先，稳定优先，价值优先。`

---

## 九、这一章读完后你要做的练习

现在请你自己完成一遍：

1. 用手机计时，做一次 60 秒 Demo 讲解
2. 再做一次 3 分钟 Demo 讲解
3. 再模拟一次“语音不稳”时的救场

如果你能把这三轮都顺利讲下来，现场表现会非常稳。
