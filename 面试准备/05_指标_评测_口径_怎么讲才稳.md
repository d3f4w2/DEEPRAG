# 05. 指标、评测、口径，怎么讲才稳

这一章非常关键。

因为很多技术面试不是被“不会”卡住，而是被“口径不稳”卡住。

你这个项目里，最容易被追问的地方就是评测。

所以你这一章要达成的目标是：

1. 记住必须会背的数字
2. 知道哪些能讲，哪些不能乱讲
3. 面对追问时能诚实又不减分

---

## 一、先记住最重要的口径原则

你对外必须把结果分成三层：

1. `最新全量实测`
2. `历史可测结果`
3. `目标值`

不要把这三者混在一起。

这是你项目面试里最重要的“防翻车原则”。

---

## 二、最新全量实测结果

当前你最稳能讲的是：

- Run Name：`optimized_ragas_40q_v2`
- 数据集：`评测/ragas评测集_40题.json`
- 题目总数：`40`
- 失败题数：`0`
- 平均 token：`16293.77`
- 平均延迟：`34562.76 ms`
- RAGAS 可用题数：`0`
- 无检索上下文题数：`40`

### 你应该怎么讲

标准说法：

`最新全量 run 我主要对外讲三个事实：第一，40 题主链路全部跑通，失败题数为 0；第二，token 和延迟是可追溯的；第三，在当前 provider 配置下，RAGAS 评分链路存在解析兼容性问题，所以这一版不拿 RAGAS 分数对外宣称。`

这句话很强，因为它：

1. 承认问题
2. 但也说明系统是能跑通的
3. 重点落在“工程闭环”而不是“漂亮分数”

## 三、历史可测结果

你还可以讲历史兼容 run：

- Run Name：`optimized_ragas_40q`
- Faithfulness：`39.86%`
- Context Recall：`0.00%`
- Answer Correctness：`0.00%`
- 平均 token：`14395.48`
- 平均延迟：`39080.38 ms`
- 题目总数：`40`
- 失败题数：`0`

### 你应该怎么讲

标准说法：

`历史兼容 run 里，我曾经测到过 `39.86%` 的 faithfulness，所以我会把它作为“曾经可测到的历史结果”来讲，但不会把它包装成当前最新成绩。`

注意这里的关键词：

1. `历史`
2. `兼容`
3. `可测`

这三个词一说，口径就稳了。

## 四、单题优化收益

这是你最适合拿来讲“优化有效”的地方。

必须记住：

### 单题对照结果

问题：

`除了 AMOLED 和 OLED 还有哪些显示类型？`

结果：

1. 工具调用轮次：`3 -> 2`
2. 检索返回字符总量：`20799 -> 8343`
3. 首轮检索体量：`8346 -> 1498`
4. 回答覆盖：`LCD/TFT -> TFT/LCD/PMOLED`

### 你应该怎么讲

标准说法：

`如果要证明我做的优化确实有收益，我更愿意引用单题对照结果。因为它口径更清晰，也更能直接说明两阶段检索把上下文体量压下来了，同时覆盖更完整。`

这句话很聪明，因为它不强行拿有问题的全量 RAGAS 说事，而是拿最能证明优化作用的数据说话。

## 五、baseline 的正确讲法

baseline 全量汇总里，你能稳讲的有：

1. 平均 token：`6177.15`
2. 平均延迟：`13721.10 ms`

### 你不能硬比的

不要说：

`我的 RAGAS 明显优于 baseline。`

因为 baseline 是旧口径，不含完整 RAGAS 三指标。

### 你可以怎么讲

标准说法：

`baseline 和当前版本在 token、延迟上是可以直接比较的，但 RAGAS 三指标不是严格同口径，所以我不会做不严谨的横比。`

这句话特别加分，因为它显得你非常严谨。

---

## 六、最危险的问题：为什么最新 RAGAS 结果不好看？

这题你一定会被问到。

### 错误回答

1. `这个不重要。`
2. `RAGAS 本来就不准。`
3. `评测器坏了。`
4. `其实我的系统很强，只是没测出来。`

这些回答都会显得你在逃避。

### 正确回答

标准说法：

`我会把这个问题拆开看。首先，最新全量 run 证明主链路是跑通的，40 题全部完成、失败题数 0。其次，RAGAS 这一轮的问题不在主链路崩掉，而是在当前 provider / judge 组合下，结构化输出解析不稳定，所以可用题数为 0。也就是说，这更像评测链路兼容性问题，而不是我拿着无效数据去夸效果。对外我会把最新全量结果、历史可测结果和目标值严格分开展示。`

这个回答的关键，是把问题定位得很清楚：

1. 主链路能跑
2. 评测链路有兼容性问题
3. 你没有混淆口径

---

## 七、如果面试官问：为什么 token 和延迟变高了？

这也是很常见的追问。

### 标准回答

`这是一个很正常的 trade-off。因为我这版优化的重点，不是单纯追求更低成本，而是先把证据约束、critic 判断和自动补检索闭环建立起来。更严格的证据要求，会带来更多检索与判断开销，所以 token 和延迟上升了。下一步最值得做的，就是在可信性已经建立的前提下，再继续压成本。`

这句话的逻辑非常好：

1. 先承认开销上升
2. 再解释为什么上升
3. 最后说明下一步优化方向

## 九、哪些话能说，哪些话不能说

### 可以说

1. `40 题全量主链路跑通，失败题数为 0。`
2. `单题检索体量下降明显。`
3. `我把评测口径分成最新结果、历史可测结果和目标值。`
4. `当前评测链路还有 provider 兼容性问题。`
5. `下一步最值得做的是成本优化和评测稳定化。`

### 不要说

1. `我的 RAGAS 已经很强。`
2. `faithfulness 39.86% 就是我当前最新成绩。`
3. `我的系统整体显著优于 baseline。`
4. `这个项目已经证明效果很好。`
5. `评测问题可以忽略。`
