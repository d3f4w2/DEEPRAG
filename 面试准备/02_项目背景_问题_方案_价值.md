# 02. 项目背景、问题、方案、价值

这一章的目标，是让你不只是“会背项目介绍”，而是真的知道：

1. 为什么这个项目值得做
2. 它解决了什么问题
3. 它和普通聊天项目有什么差别
4. 面试官为什么会觉得它有含金量

---

## 一、项目背景

如果面试官问：

`你为什么会想到做这个项目？`

不要回答：

`因为 RAG 很火。`

也不要回答：

`因为我想做个 AI 项目放简历。`

更好的回答是：

`我关注到传统 RAG 很多时候只是“检索相似片段，再让模型总结”，这种方式在简单问答上没问题，但当问题变成否定、比较、多跳推理、跨文档聚合时，就容易不稳。同时，上下文很容易越来越大，成本也会上去。所以我想做一个不只是“能搜到”，而是“能基于结构和证据来回答”的系统。`

这里你要让面试官感觉到：

1. 你是先看到了问题，再做项目
2. 你不是随便追热点
3. 你有“问题意识”

---

## 二、传统 RAG 的几个典型问题

你一定要熟练说出下面这些点。

### 1. 文档被切碎后，结构丢失

传统 RAG 会把文档切成 chunk，但文件层级、章节关系、目录关系会弱化。

影响：

1. 模型更难建立全局视角
2. 更难回答“哪个产品线最贵”“除了 A 还有什么”这种问题

### 2. 相似召回不等于正确推理

传统 RAG 的核心是“相似度高就召回”，但很多问题不是找相似，而是找：

1. 反例
2. 对比项
3. 排除项
4. 全局最值

比如：

`除了 AMOLED 和 OLED 还有哪些显示类型？`

这个问题其实不是“找最像 AMOLED/OLED 的内容”，而是“做集合排除”。

### 3. 上下文容易变大

如果每轮都整文件召回，或者一召回就很多 chunk，成本会上升。

问题不只是钱，还包括：

1. 延迟更高
2. 模型更容易被无关信息干扰
3. 可解释性变差

### 4. 没有证据约束时，模型会“顺着说”

模型有时会生成看起来合理的答案，但未必有证据。

这就是你项目里“证据放行”特别重要的背景。

## 三、你的方案到底解决了什么

你的项目不是单点优化，而是一个完整链路。

### 1. 用 Deep RAG 的思路保留结构

你不是先把所有东西切碎，而是让模型先有“知识地图”。

这意味着模型知道：

1. 有哪些文件
2. 文件之间是什么关系
3. 可以从哪里进一步取证

### 2. 用两阶段检索控制上下文

第一步 `search_paths`：

1. 从摘要和路径中筛候选文件
2. 返回的是候选范围，不是整文件正文

第二步 `retrieve_sections`：

1. 从候选文件里抽取证据段落
2. 真正送给模型的是“片段级证据”

结果：

1. 上下文变小
2. 证据更集中
3. 检索过程更可解释

### 3. 用证据池 + critic 控制回答权限

你的系统不是“模型一想到答案就结束”，而是：

1. 先检索
2. 再整理证据
3. 再判断证据够不够
4. 够了才回答
5. 不够就继续检索或者拒答

这一步特别能体现你的工程意识。

### 4. 用评测闭环约束“讲得好不好看”

你不是只看页面效果，而是引入：

1. token
2. 延迟
3. RAGAS
4. 基线对比

这说明你在做“系统优化”，不是“页面演示”。

### 5. 用多模态入库扩展系统边界

语音和图片入库这部分的价值，不只是“多了两个页面”，而是：

1. 新信息可以快速进入知识库
2. 仍然复用统一检索链路
3. 系统扩展性更强

## 四、这个项目的价值应该怎么说

面试里，价值不要说得太虚。

你可以分成三个层次说。

### 1. 用户价值

`用户不只是拿到答案，而是拿到带证据的答案。`

### 2. 工程价值

`系统不是一堆 prompt 拼起来，而是有检索、证据、评测、预算、入库这些完整链路。`

### 3. 成长价值

`这个项目让我真正经历了从想法、设计、实现、评测到口径管理的完整过程。`

## 五、为什么这个项目适合实习面试

因为它同时具备下面几个特点：

1. 有 AI 背景，但不是只会调 API
2. 有后端逻辑，有前端展示，有评测，有工程细节
3. 既能讲技术，也能讲取舍
4. 有结果，有问题，也有下一步方向

实习项目最怕两种情况：

1. 太简单，像课程作业
2. 太大，讲不清自己做了什么

你的这个项目，刚好适合讲成：

`我不是从零发明 RAG，但我把 Deep RAG 思路做成了一个更工程化、更适合展示的系统，并且我能说清楚自己在里面亲自解决了哪些问题。`

---

## 六、不要这样讲

下面这些话很危险：

1. `这个项目的指标已经非常强了。`
2. `这个项目已经完全解决了 RAG 幻觉问题。`
3. `这个系统比传统 RAG 全面更好。`
4. `RAGAS 现在已经证明我的系统效果很强。`

为什么危险：

1. 容易被抓住口径问题
2. 容易被追问更严谨的数据
3. 容易显得夸张

更稳的说法是：

1. `我验证了链路可以跑通，也验证了单题检索体量能下降。`
2. `我把无证据回答的风险收紧了，但不是说彻底消灭幻觉。`
3. `我把当前结果、历史可测结果、目标值分开呈现，避免口径混淆。`
