# 07. 技术追问标准回答库

这一章是你的“技术防御体系”。

使用方法：

1. 不要试图逐字背诵
2. 先记住回答结构
3. 再用自己的语言复述

建议你每个问题都按下面结构回答：

1. 先一句话结论

## A. RAG 与 Deep RAG 基础

### 1. 什么是 Deep RAG？

标准回答：

`我理解的 Deep RAG，不是简单的 chunk 检索，而是尽可能保留知识库的结构，让模型先知道“知识地图”在哪里，再按需检索和导航。它更强调结构、路径和多轮取证，而不是一次性相似召回。`

加分点：

1. 你理解的是“结构化导航”
2. 不是只会说“更高级的 RAG”

### 2. 你的项目和传统 RAG 最大区别是什么？

标准回答：

`最大的区别是，我不是直接切 chunk 然后相似召回，而是先做候选路径检索，再做证据片段提取，并把回答放到证据约束之下。也就是说，我更强调检索路径和证据质量，而不是只追求召回一堆相似文本。`

### 3. 为什么传统 RAG 在复杂问题上容易不稳？

标准回答：

`因为传统 RAG 的核心是相似召回，但很多复杂问题不是找相似，而是做比较、排除、聚合或者跨文档推理。如果知识被切碎后只靠相似度排序，就可能漏掉真正需要的上下文关系。`

---

## B. 两阶段检索设计

### 4. 为什么要拆成 `search_paths -> retrieve_sections`？

标准回答：

`因为这两个阶段解决的是不同问题。`search_paths` 负责快速缩小范围，`retrieve_sections` 负责从缩小后的范围里取高价值证据。拆开以后，既能控成本，也能提高证据密度。`

### 5. `search_paths` 具体在做什么？

标准回答：

`它主要基于文件路径和摘要做轻量候选检索，返回候选路径、分数和摘要，而不是正文。这一步的目标是先决定“去哪些文件里找”，而不是“马上把文件内容全喂进去”。`

### 6. `retrieve_sections` 具体在做什么？

标准回答：

`它会对候选文件按 Markdown 标题切段，再按 query 相关性排序，提取有限数量的高相关片段作为证据。这样模型看到的是更聚焦的证据，而不是整篇内容。`

### 7. 为什么不直接用向量库做端到端召回？

标准回答：

`可以做，但这个项目里我希望保留文件结构和路径信息，同时增强可解释性，所以没有把核心逻辑完全交给黑盒式向量召回。我的设计里，路径和摘要本身就是重要的检索线索。`

边界补充：

`后续如果要继续优化，也可以把更强的向量信号融进候选检索，但当前版本优先的是结构和工程可控性。`

## C. 证据与幻觉控制

### 8. 什么是 evidence_pool？

标准回答：

`evidence_pool 是检索后形成的结构化证据集合，里面保存了文件路径和对应片段。它的作用是把“检索结果”变成“回答依据”，后续的回答和 judge 判断都围绕它展开。`

### 9. 为什么回答必须带 `### 证据`？

标准回答：

`因为我希望回答结果是可追溯的。带 `### 证据` 不只是为了展示格式，而是为了明确回答引用了哪些文件和片段，避免答案变成无法核对的自然语言生成。`

### 10. critic 的作用是什么？

标准回答：

`critic 的作用是判断当前证据是否足够支持回答。它会在 accept、revise、refuse 之间做判断，从而决定是继续检索、允许回答，还是拒答。`

### 11. 这是不是就完全解决幻觉了？

标准回答：

`不能说完全解决。更准确地说，我把“无证据也回答”的风险显著收紧了，让回答更依赖证据、更容易核对，但模型生成系统本身仍然不可能通过这一层就彻底消灭幻觉。`

这题一定要诚实。

## D. 评测与指标

### 12. 你用了哪些评测指标？

标准回答：

`我主要看三类东西。第一类是链路是否跑通，比如 40 题全量完成情况；第二类是成本性能，比如平均 token 和平均延迟；第三类是历史兼容 run 下可测的 RAGAS 指标，比如 faithfulness。`

### 13. 为什么你不直接强调 RAGAS 分数？

标准回答：

`因为我这个项目当前的最新全量 run 在 provider / judge 组合下有 RAGAS 解析兼容性问题，所以我不想拿不稳的数据做强结论。对外我会把最新全量结果、历史可测结果和目标值分开。`

### 14. 为什么优化后 token 和延迟变高？

标准回答：

`因为这版优化优先建立了证据约束和补检索闭环，更严格的证据要求本身就会带来更多检索和判断开销。也就是说，这是一种可信性优先的 trade-off，下一步才是继续压成本。`

### 15. 你最稳能讲的结果是什么？

标准回答：

`最稳的是最新全量 run：40 题全部跑通，失败题数 0，平均 token 和平均延迟都有可追溯产物。其次是单题对照里检索体量下降明显。`

## E. 多模态与扩展性

### 16. 为什么要做语音和图片入库？

标准回答：

`因为我不希望系统只是一个静态知识库问答器。我希望它具备持续吸收新知识的能力，而语音和图片正是很常见的新信息来源。`

### 17. 为什么要用 draft + ingest 两阶段？

标准回答：

`因为多模态原始输入很容易有噪声，尤其语音识别和图片 OCR。先 draft，再由人确认后 ingest，可以平衡效率和数据质量。`

### 18. 入库后为什么能马上检索到？

标准回答：

`因为写入知识库之后，我会同步更新 `summary_demo.json` 的索引内容，并清理缓存状态，所以后续 `search_paths` 能立刻看到新内容。`

### 19. 语音和图片为什么不走独立问答逻辑？

标准回答：

`因为我更想要一个统一系统，而不是多个孤立功能。语音和图片最后都会转成可检索知识文档，复用同一条检索链路，这样系统更一致，也更容易维护。`

## F. 工程实现与稳定性

### 20. 为什么选 FastAPI？

标准回答：

`因为这个项目核心是接口编排、异步调用和流式响应，FastAPI 在 Python 生态里比较适合这类 AI 后端场景，开发速度和结构清晰度都比较好。`

### 21. 为什么选 React + TypeScript？

标准回答：

`因为我需要一个可交互、可视化工具链路和证据轨迹的前端。React 适合状态驱动的界面，TypeScript 能提高类型约束和可维护性。`

### 22. 你做过哪些工程收尾？

标准回答：

`我做过一轮比较完整的收尾，包括前端 lint / build 打通、pytest 入口稳定化、语音页高频更新稳定性修复，以及评测口径和文档一致性整理。`

### 23. 项目里你印象最深的 bug 是什么？

标准回答：

`语音页开始录音后出现灰屏，是我印象比较深的一个问题。它不是简单语法错，而是高频状态更新、组件切换和开发态行为叠加导致的稳定性问题。最后我是通过收紧回调保护、降低重渲染和加错误边界来修掉的。`

## G. 取舍与下一步

### 24. 如果继续做，你下一步会优化什么？

标准回答：

`我会优先做两件事。第一，把当前 provider 下的 RAGAS 兼容性问题稳定下来，让评测口径更完整；第二，在证据约束已经建立的前提下，继续压 token 和延迟。`

### 25. 如果让你重做一次，你会改什么？

标准回答：

`如果重做一次，我会更早把评测口径和基线定义清楚，这样后面做对比时会更顺。另外，我会更早把检索和证据标准化，而不是先做页面展示。`

### 26. 这个项目最大的 trade-off 是什么？

标准回答：

`最大的 trade-off 是可信性和成本之间的平衡。证据约束越严格，系统越稳，但检索和判断开销也会增加。我当前版本更偏向先把可信性闭环立住。`

## 最后给你一个万能回答模板

很多追问其实都可以用这个模板：

`我当时遇到的问题是 X。直接做 Y 虽然更简单，但会带来 Z，所以我最后选择了 A 方案。这个方案的结果是 B，不过它也带来了 C 这个 trade-off，这也是我下一步准备继续优化的地方。`

这就是成熟工程师最常见的说话方式。

你把这个模板用熟，技术面会非常加分。
