# 01. 先把项目讲清楚

这份文档的目标很简单：

读完以后，你必须能把项目用四种长度讲清楚：

1. 10 秒版
2. 30 秒版
3. 90 秒版
4. 3 分钟版

如果这一步做不好，后面所有技术细节都会显得零散。

---

## 一句话版本

先背这一句：

`我做了一个基于 Deep RAG 思路的知识问答系统，把传统“整段召回”升级成“候选路径 + 证据片段”的检索链路，并加入证据放行、评测闭环和语音/图片入库，让回答更可追溯、更工程化。`

这句话基本覆盖了：

1. 项目类型：知识问答 / RAG
2. 核心特色：Deep RAG
3. 你的优化：两阶段检索
4. 工程亮点：证据放行、评测、多模态
5. 最终价值：可追溯、工程化

---

## 10 秒版

适用场景：

1. 面试官刚问“你最近做了什么项目”
2. 你要先开个头
3. 面试官明显想快速了解，不想听太长

标准说法：

`我做了一个 Deep RAG 知识问答项目，重点把检索做成了“候选路径 + 证据片段”，并加了证据放行和评测闭环，让回答不只是能答，还要有证据。`

这 10 秒里，你只要把下面三个词打出去就够了：

1. `Deep RAG`
2. `候选路径 + 证据片段`
3. `有证据的回答`

---

## 30 秒版

适用场景：

1. 面试官说“展开讲讲”
2. 一面开场自我介绍后你要介绍项目
3. 技术面第一问

标准说法：

`这个项目本质上是一个知识库问答系统，但我没有停留在传统 RAG 的“切 chunk 然后相似召回”上，而是按 Deep RAG 的思路去做。具体来说，我把检索链路从整文件多轮召回，改成了先 `search_paths` 找候选文件，再 `retrieve_sections` 抽证据片段，这样上下文更可控。然后我又加了证据池放行和 critic 判断，要求回答必须带 `### 证据`。最后，我还把语音和图片入库打通，并用评测去看 token、延迟和 RAGAS 指标，让项目更像一个完整的工程系统。`

你会发现，这个版本天然是“三段式”：

1. 原问题：传统 RAG 不够好
2. 你的方案：两阶段检索 + 证据约束
3. 工程闭环：多模态 + 评测

---

## 90 秒版

这是最重要的版本。

因为技术面里，最常见的节奏就是：

1. 你先说 1 分钟左右
2. 面试官再从里面挑点深挖

标准说法：

`我这个项目是一个基于 Deep RAG 思路做的知识问答系统。传统 RAG 一般会先把文档切成 chunk，再做相似度召回，但这类方法在否定关系、比较、跨文档推理、全局聚合这种问题上比较吃力，而且上下文很容易越喂越大。`

`所以我做的第一个核心优化，是把检索链路改成两阶段：先用 `search_paths` 基于路径和摘要做轻量候选检索，再用 `retrieve_sections` 从候选文件里抽取高相关证据片段，而不是整文件塞给模型。这样单题实测里，检索返回字符总量从 20799 降到了 8343，首轮检索体量从 8346 降到 1498。`

`第二个核心优化，是我把“回答权限”收紧了。系统不是模型想答就答，而是先形成 evidence_pool，然后通过 critic 判断证据是否足够，最终答案必须带 `### 证据`。证据不足就自动继续检索，必要时拒答。`

`第三个部分是工程化补全。我把语音和图片做成 draft + ingest 双阶段入库，入库后会自动更新索引，所以新增内容下一轮就能被统一检索链路命中。然后我还补了评测闭环，能看 40 题全量跑通、平均 token、平均延迟，以及历史兼容 run 的 faithfulness。整体上，这个项目的价值不是“做了个聊天页面”，而是把一个可追溯、可解释、可扩展的 RAG 系统真正做成了。`

---

## 3 分钟版

这个版本用于：

1. 面试官明确说“详细讲讲”
2. 主管面 / 二面
3. 你想把节奏掌握在自己手里

标准结构如下：

### 第一段：先讲问题

你先说：

`我做这个项目的出发点，是我觉得传统 RAG 很容易停留在“检索相似文本，再把文本喂给模型”的层面。它能回答一部分问题，但对于比较复杂的查询，比如否定关系、跨文档比较、多跳推理、全局聚合，其实效果并不稳定。另一个问题是，它很容易把上下文越喂越大，导致 token 成本上升。`

### 第二段：再讲核心方案

你接着说：

`所以我采用 Deep RAG 的思路，把知识库结构保留下来，让模型先看到知识地图。工程上，我把主链路设计成 `search_paths -> retrieve_sections`。第一步用摘要和路径做候选检索，第二步只从候选文件中抽相关片段。这样既保留结构，又控制上下文。`

### 第三段：讲可信性

你继续说：

`为了降低幻觉，我做了证据池放行机制。模型生成答案前，系统会先把证据片段结构化，形成 evidence_pool。最终答案要求必须带 `### 证据`。如果当前证据不足，critic 会判定继续检索，或者干脆拒答。这个设计的本质，是把回答从“自由生成”改成“证据驱动生成”。`

### 第四段：讲工程化

你再补：

`我不想让它只是一个 demo，所以继续补了工程闭环。一个是多模态，把语音和图片做成 draft + ingest 两阶段，人确认后再入库，入库后会自动更新 summary 索引，保证新知识下次可检索。另一个是评测，我做了 40 题全量评测、token 和延迟统计，并保留历史兼容 run 的可测结果。`

### 第五段：讲结果

最后收口：

`从结果上看，这套链路至少证明了三个事情：第一，40 题全量主链路能稳定跑通，失败题数是 0；第二，单题检索体量确实下降了，说明检索更精细；第三，整个系统具备继续优化的空间，比如成本和 RAGAS 兼容性问题都已经被我明确识别出来了。对我来说，这个项目最大的价值不是“做完了一个功能”，而是我把一个技术想法做成了一个可以讲清楚、跑得通、还能继续迭代的工程项目。`

---

## 面试官最想从这段里听到什么

你说这段话时，面试官其实在判断下面几件事：

1. 你有没有真实理解 RAG，而不是只会调用模型 API。
2. 你有没有发现问题并自己设计方案。
3. 你是不是只会“堆功能”，还是能讲清取舍。
4. 你有没有工程意识，比如稳定性、指标、可追溯性。

所以你一定要把重点放在：

1. 为什么要这么设计
2. 你改了什么
3. 结果是什么
4. 哪些问题还没完全解决

---

## 你现在就要开口练

请你现在做三件事：

1. 用 10 秒说一遍项目。
2. 用 30 秒说一遍项目。
3. 用 90 秒说一遍项目。

如果你说的时候出现下面情况，说明还不够熟：

1. 说了一堆名词，但没有“问题 -> 方案 -> 结果”
2. 一上来就在讲前端页面
3. 指标说不清
4. 总在说“这个功能、那个功能”，却没说价值

4. 我做了 `语音/图片入库 + 评测闭环`，让它更工程化。
