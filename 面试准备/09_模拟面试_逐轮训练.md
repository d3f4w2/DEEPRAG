# 09. 模拟面试：逐轮训练

这一章不是单纯给你看答案，而是要你真的练。

建议做法：

1. 先只看问题，不看参考回答
2. 自己回答 30 到 90 秒
3. 再看参考思路补强

---

## 第一轮：开场与项目介绍

### 题 1：你先简单介绍一下这个项目

你应该答到：

1. 这是一个 Deep RAG 知识问答项目
2. 核心是两阶段检索
3. 有证据放行
4. 有评测和多模态入库

### 题 2：你为什么会做这个项目？

你应该答到：

1. 传统 RAG 的不足
2. 你不想做纯 API Demo
3. 你想做工程化闭环

### 题 3：这个项目里你最核心的贡献是什么？

你应该答到：

1. 两阶段检索
2. 证据池放行
3. 评测与口径管理

---

## 第二轮：技术细节深挖

### 题 4：为什么要把检索拆成两阶段？

你至少要说出：

1. `缩小范围`
2. `抽高价值证据`
3. `控制 token`

### 题 5：你怎么降低幻觉？

你至少要说出：

1. evidence_pool
2. critic
3. `### 证据`
4. 不足就补检索或拒答

### 题 6：为什么最新结果里 RAGAS 不好看？

你必须说出：

1. 最新 run 主链路是通的
2. 问题在评测兼容性
3. 口径必须分开

### 题 7：为什么 token 和延迟变高了？

你必须说出：

1. 可信性更严格
2. 自动补检索有成本
3. 这是 trade-off

### 题 8：多模态为什么不是单独系统？

你必须说出：

1. 统一知识入口
2. 统一检索链路
3. 更容易维护和解释

---

## 第三轮：工程意识深挖

### 题 9：你最后为什么还做 lint、pytest、文档口径这些？

你应该答到：

1. 项目要能讲、能交付
2. 不只是功能做出来
3. 工程收尾能体现成熟度

### 题 10：如果让你继续做，你下一步做什么？

你应该答到：

1. 修评测兼容性
2. 压 token / 延迟
3. 继续打磨可交付性

### 题 11：你项目里最难的一次问题定位是什么？

建议答语音灰屏，结构如下：

1. 现象
2. 根因
3. 修复思路
4. 结果

---

## 第四轮：压力追问

这一轮故意更尖锐，你必须练。

### 题 12：你这个项目是不是其实就是普通 RAG 加了点包装？

参考思路：

`我觉得不能这么概括。它当然仍然属于 RAG 系统，但我做的重点不是包装，而是把检索链路、证据约束和评测闭环补完整。也就是说，它不是“换个页面”，而是回答机制和工程结构都发生了变化。`

### 题 13：你现在的指标并不好看，为什么还说项目不错？

参考思路：

`因为我不会只看单点分数。我会区分“主链路能否跑通”“单题优化是否有效”“评测链路是否稳定”这三件事。当前主链路已经跑通，单题优化收益也有明确对照，评测兼容性问题我也如实说明了。`

### 题 14：你怎么证明这些是你自己做的？

参考思路：

`我可以把任一核心模块拆成问题、动作、结果来讲，比如两阶段检索、critic 或多模态入库。如果您愿意，我可以直接挑一个展开到实现层。`

### 题 15：如果我是用户，我为什么要用你的系统？

参考思路：

`因为它给的不只是一个答案，而是一个带来源和证据的答案。尤其在知识库场景里，这比单纯“能回答”更重要。`

---

## 第五轮：你现在就开口模拟

请你真的按下面顺序练一遍：

1. 用 30 秒回答题 1
2. 用 60 秒回答题 4
3. 用 60 秒回答题 6
4. 用 60 秒回答题 13
5. 用 90 秒回答题 11

如果你这一轮下来能基本流畅，面试状态会非常接近“可上场”。
