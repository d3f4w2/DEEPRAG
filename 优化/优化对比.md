﻿﻿# Deep RAG 优化对比（复习版）

## 0. 本次做过的三类优化（你要讲的主线）

1. **两阶段检索降 token**
   - 从“整文件读”改成“先定候选文件，再抽证据片段”。
2. **防幻觉：证据片段 + 证据池放行 + 停检判断**
   - 答案必须带 `### 证据`，并经过证据池校验，不满足就继续检索。
3. **评测升级：RAGAS + 性能**
   - 指标纳入忠诚度、上下文召回率、答案质量（当前代码字段名是`答案准确度`）、平均 token、平均延迟。

---

## 1. 优化点一：两阶段检索（核心为控 token）

### 1.1 改造前后链路

- 改造前：`retrieve_files -> retrieve_files -> retrieve_files`（整文件注入）
- 改造后：`search_paths -> retrieve_sections`（候选范围 + 段落级证据）

### 1.2 具体做法（实现细节）

#### A. 候选路径检索：`search_paths`
- 基于 `summary_demo.json` / 文件路径建立轻量索引。
- 词法匹配打分（英文 token + 中文 bi-gram）。
- 返回 `top_k` 候选路径 + 分数 + 摘要（不是整文件正文）。
- `top_k`被安全限制（默认上限 10），避免候选过宽。

#### B. 片段检索：`retrieve_sections`
- 对候选文件按 Markdown 标题切段（无标题则按固定块兜底切段）。
- 段落按 query 相关性排序，只取每文件前 N 段（默认 2，最大 4）。
- 每段输出标准化证据头：
  - `[[FILE:... | SECTION:... | SCORE:... | HEADING:...]]`
- 片段级限流（防 token 爆炸）：
  - 单段长度上限：`MAX_SECTION_CHARS`（默认 1200）
  - 总段落字符上限：`MAX_SECTION_TOTAL_CHARS`（默认 9000）

#### C. 编排层强约束（`main.py`）
- 仅允许工具：`search_paths`、`retrieve_sections`。
- 强制先走 `search_paths`，后走 `retrieve_sections`。
- 若 `retrieve_sections` 文件数不足，会自动补齐到 `min_file_paths`（默认 4）提升覆盖。
- 有一个轻量 router（LLM）按问题类型动态给 `top_k / max_sections_per_file / min_file_paths`。

### 1.3 单题实测对比（已落盘）

**问题：** `除了 AMOLED 和 OLED 还有哪些显示类型？`

| 指标 | 优化前 | 优化后 | 变化 |
|---|---:|---:|---:|
| 工具调用轮次 | 3 | 2 | -33.3% |
| 工具调用链路 | `retrieve_files ×3` | `search_paths -> retrieve_sections` | 两阶段替代整文件检索 |
| 工具返回字符总量（近似上下文体量） | 20,799 | 8,343 | **-59.9%** |
| 首轮检索体量 | 8,346 | 1,498 | **-82.1%** |
| 回答覆盖 | LCD/TFT（并提示可能不全） | TFT/LCD/PMOLED（并区分 AMOLED 增强型） | 证据覆盖更完整 |

优化后 usage（流式统计）：
- input tokens：`6110`
- output tokens：`237`
- 分轮：`1226/36`、`1643/117`、`3241/84`（in/out）

## 2. 优化点二：防幻觉机制（证据池放行 + 停止检索）

### 2.1 证据片段标准化

- 从 `retrieve_sections` 的输出中用正则提取证据块。
- 形成结构化证据项：`file_path + snippet`。
- 证据片段超长会截断，且去重合并进 `evidence_pool`。

### 2.2 证据池放行机制（硬约束）

- 最终答案统一走格式化器：
  - 必带 `### 证据`
  - 默认最多展示 3 条证据项
- 如果当前回答没有证据池支撑：
  1. 发出 `retrieval_judge: stop=false` 事件（证据不足）
  2. 自动追加一轮（或多轮）`search_paths + retrieve_sections`
  3. 仍无证据则返回兜底拒答（避免无证据胡答）

### 2.3 LLM 判断“是否可停检索”

- 在 ReAct 模式里引入 `_llm_should_stop_retrieval`：
  - 输入：问题 + 候选答案 + 证据池预览
  - 输出：严格 JSON `{"stop": bool, "reason": string}`
- 只有判定证据足够时才停止检索并出最终答案；
- 否则继续自动检索（最多 `MAX_AUTO_RETRIEVAL_ROUNDS=3`）。

> 这套机制本质是把“回答权限”从模型自由生成，改成“证据池放行后才能回答”，专门用来压幻觉。

## 3. 优化点三：评测升级（RAGAS + token/耗时）

### 3.1 评测脚本口径（`评测/运行小评测.py`）

- RAGAS 指标：
  - `faithfulness`（忠诚度）
  - `context_recall`（上下文召回率）
  - `answer_correctness`（代码字段名：`答案准确度`；当前实现用它来承接你说的“答案相关性/答案质量”口径）
- 性能指标：
  - `平均token`（由流式 `usage` 累加后按题均值）
  - `平均延迟_ms`（每题请求到 done 的耗时均值）
- 结果文件：
  - `*_明细.json`（题级）
  - `*_汇总.json`（总体）
  - `*_报告.md`（可读报告）
- 支持 `--compare` 直接输出“当前 vs 基线”差值表。

### 3.2 评测管理器联动

- `evaluation_manager.py` 会读取上述指标并做任务级失败判定：
  - 忠诚度 / 上下文召回率 / 答案准确度低于阈值（默认 0.75）即标记失败题。
- 前端展示也已对齐这套指标字段。

### 3.3 当前仓库里已有评测结果情况

- 已有完整 40 题基线：`评测/结果/baseline_40q_汇总.json`
  - 平均 token：`6177.15`
  - 平均延迟：`13721.10 ms`
- 单题“优化前后”检索链路对比与 token 体量：见第 1.3 节（来自 `优化/*.json`）。
- 若要补全“40题优化后 vs baseline_40q”的RAGAS对比，可执行：
  - `python 评测/运行小评测.py --run-name optimized_40q --compare 评测/结果/baseline_40q_汇总.json`

## 4. 你面试可直接说的总结

1. **检索策略层**：从整文件召回改为“候选路径 + 段落证据”，把上下文注入量压下来。  
2. **可信性层**：答案必须通过证据池放行；证据不足时自动继续检索，必要时拒答。  
3. **评测层**：用 RAGAS（忠诚度/上下文召回/答案质量）+ 成本性能（token/延迟）做闭环，支持基线对比。

## 6. 优化点四：语音实时入库（多模态第一阶段）

### 6.1 目标与边界
- 目标：把“实时语音 -> 文本知识 -> 可检索证据”打通，作为多模态第一阶段落地。
- 边界：当前先做语音，不做图片；语音场景优先“会议/口述信息沉淀”。

### 6.2 前端：独立语音入库界面（单开页面）
- 新增导航页：`语音入库`（与 Chat / 测评并列）。
- 页面能力：
  - 实时语音识别（浏览器 `SpeechRecognition` / `webkitSpeechRecognition`）。
  - 实时显示原始转写文本。
  - 实时触发 LLM 草稿生成（润色文本 + 核心摘要）。
  - 用户可编辑“确认入库文本/摘要”，并手动“同步最新 AI 草稿”。
  - 只有用户点击“确认入库”才会真正写入知识库（人工放行）。

### 6.3 后端：语音两段式接口
- `POST /voice/draft`
  - 输入：`transcript`、`author`、`provider`（可选）。
  - 输出：`polished_text`、`summary`、`warning`（可选）。
  - 作用：将实时转写内容整理为可读文本，并抽取一句核心语义摘要。
- `POST /voice/ingest`
  - 输入：`transcript`、`summary`、`author`、`source`、`occurred_at`、`raw_transcript`。
  - 输出：`status`、`file_path`、`created_at`。
  - 作用：仅在用户确认后执行落库。

### 6.4 入库格式（可追溯）
- 入库目录：`Knowledge-Base/Voice-Notes/YYYY-MM-DD/`
- 文件名：`HHMMSS_author.md`（自动去重后缀）
- 内容字段：
  - 时间戳（Timestamp）
  - 作者（Author）
  - 来源（Source）
  - 语音摘要（Summary）
  - 纠正后文本（Corrected Transcript）
  - 原始转写（Raw Transcript，可选）

### 6.5 检索可用性联动
- 语音文件写入后，同步更新 `summary_demo.json` 的路径摘要索引。
- 这样 `search_paths -> retrieve_sections` 可直接覆盖新入库语音笔记，保证“刚入库即可被检索链路命中”。

### 6.6 面试可讲价值
1. **多模态扩展路径清晰**：先语音后图片，分阶段验证，降低一次性复杂度。
2. **人机协作闭环**：机器实时草稿，人工最终确认，兼顾效率与可控性。
3. **知识可追溯**：每条语音知识都带作者、时间戳、来源与摘要，后续审计与复盘成本低。

## 7. 优化点五：语音页面稳定性修复（解决开始录音后灰屏）

### 7.1 问题现象
- 点击“开始录音”后，前端出现灰屏。
- 控制台报错：`NotFoundError: Failed to execute 'insertBefore' on 'Node'`。

### 7.2 根因与处理
- 根因：录音期间高频状态更新 + 组件频繁切换，导致 React 在开发模式下出现 DOM 插入时序冲突。
- 处理动作：
  - 语音识别回调 `onresult` 增加空值保护与异常兜底，避免回调数据结构异常导致运行时崩溃。
  - 录音按钮改为“单按钮同树渲染”，避免 `Mic/MicOff` 反复切换造成节点重排风险。
  - 自动草稿请求改为静默防抖，降低录音过程中的重渲染频率。
  - 移除语音页动态 loading 图标分支，减少高频文本节点插拔。
  - 增加 `VoiceErrorBoundary`，即使异常也能降级展示而非整页失效。
  - 开发态去掉 `StrictMode`，避免语音相关副作用双执行放大不稳定性。
  - `index.html` 增加 `notranslate` 元信息，降低浏览器翻译插件改写 DOM 的概率。

### 7.3 当前结果
- 语音页具备异常隔离能力，出现异常时可恢复重试。
- 录音流程在开发环境稳定性显著提升。

## 8. 优化点六：入库后可检索性增强（确保“下次能搜到”）

### 8.1 机制升级
- 入库成功后不只写文件，还会自动更新 `summary_demo.json` 的索引条目。
- 索引内容从“仅摘要”增强为：`摘要 + author + source + date/time + transcript片段`。

### 8.2 为什么更容易命中
- `search_paths` 采用词法匹配（路径 + 摘要语料），扩展后的索引字段可覆盖更多问法：
  - 按作者问（如“Unknown 说了什么”）
  - 按来源问（如“Realtime microphone 的记录”）
  - 按日期/时间问
  - 按正文关键词问

### 8.3 与检索链路的联动
- `update_summary_entry` 更新索引后会清理内存缓存（`_summary_index_cache = None`），下一次检索立刻生效。
- 最终仍走统一链路：`search_paths -> retrieve_sections`，保证语音知识与原有知识库检索方式一致。

### 8.4 可操作验收方式
1. 入库后在 `summary_demo.json` 确认新增语音文件路径及索引文本。
2. 在聊天中按“作者/来源/关键词/日期”分别提问，验证 `search_paths` 是否包含该语音文件路径。
3. 查看回答中的 `### 证据` 是否引用该语音 Markdown 文件。
