关键：大文件vs小文件
                    token 数
                      │
              3000 ───┼──────────── 大文件分界线
                      │
         ┌────────────┴────────────┐
         │                         │
    ▼ 小文件                     ▼ 大文件
    直接丢给 LLM：               需要先拆分：
    "请总结这个文件"              "请把这个文件切成N块，每块给个摘要"

    ┌─────────────┐              原始文件 (12000 token)
    │ LLM 输入    │                    │
    │ 整个文件内容  │                    │ 拆分
    │             │                    │
    │ LLM 输出    │              ┌─────┼─────┐
    │ "一句话摘要" │              ▼     ▼     ▼
    └─────────────┘           chunk1  chunk2  chunk3
                             1-63行   64-120行 121-200行
                                │      │        │
                                ▼      ▼        ▼
                             各自的摘要字符串

    最终结果:                 最终结果:
    "字符串"                  [ {"start":1, "end":63,
                                "summary":"..."},
                                {"start":64, "end":120,
                                "summary":"..."},
                                ... ]
扫描知识库所有 .md 文件 → 用 LLM 给每个文件生成极短摘要 → 大文件先拆块再分别摘要 → 最终输出一棵带摘要的目录树

它的目的是：把几万 token 的知识库压缩成几百 token 的目录树摘要，
让主聊天程序（你上一段代码里的 knowledge_base.get_file_summary()）能快速判断"用户的问题该去读哪个文件"，而不用每次都把所有文件塞给 LLM。
